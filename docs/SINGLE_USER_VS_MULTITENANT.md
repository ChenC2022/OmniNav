# 单用户 vs 多租户：方案差别与难易程度

本文档仅做对比说明，不修改《安全性改进方案》中的既定计划。便于在选型时理解两种模式在改动范围、难度和可行性上的差异。

---

## 一、概念区分

| 维度 | 单用户（当前目标） | 多租户 |
|------|-------------------|--------|
| **含义** | 一个部署只服务一个“主人”，访问者即主人本人（或通过门禁只放行主人）。 | 一个部署服务多个独立用户，每人一套书签/分类/设置，数据互不可见。 |
| **典型场景** | 个人自用、家庭/小团队共用一个“门禁”（如同一 Access 策略）。 | 多人共用同一域名或同一应用，每人登录后只看自己的数据。 |
| **“用户”在系统中的存在感** | 无“用户账号”概念，或仅有一个“主人”身份（用于鉴权，不参与数据分片）。 | 有明确用户标识（如 userId），所有数据按用户隔离。 |

---

## 二、单用户方案（对外暴露时）

### 2.1 数据与存储

- **KV 结构**：无需改动。继续使用全局 key：`bookmarks`、`categories`、`pinned`、`settings`。
- **谁可以访问**：通过“门禁”或“主人密码”保证只有主人能访问整站或敏感 API，数据本身不分片。

### 2.2 认证方式（二选一）

| 方式 | 说明 | 应用内改动 |
|------|------|------------|
| **Cloudflare Access** | 在边缘做门禁（如 One-time PIN、Google 登录），通过后才到 Pages/API。 | **无**。仅 Dashboard 配置。 |
| **应用内主人密码** | 在 Worker 中校验密码（wrangler secret），通过后发 cookie/JWT；所有敏感 API 检查该 cookie/JWT。 | 需新增：登录接口、鉴权中间件、前端登录页与 401 处理。 |

### 2.3 需要改动的部分（仅“应用内主人密码”时）

- **后端**  
  - 新增 `POST /api/auth/login`（校验密码，写 cookie 或返回 JWT）。  
  - 鉴权中间件：对 `/api/data/*`、`/api/admin/*`、`/api/ai/*` 校验 cookie/JWT，无则 401。  
  - 可选：`POST /api/auth/logout`（清除 cookie 或使 token 失效）。
- **前端**  
  - 登录页：未带有效会话时展示，提交密码调 login，成功后跳转/进入主界面。  
  - 请求：同源下 cookie 自动携带；若用 JWT 需在 header 中带 token。  
  - 401 处理：未登录或过期时跳回登录页或提示重新登录。
- **配置**  
  - wrangler secret：`OMNINAV_OWNER_PASSWORD`。  
  - 登录限流（防暴力破解）：边缘 Rate Limiting 或应用内按 IP 计数（如用 KV）。

### 2.4 难度与可行性

| 项目 | 评估 |
|------|------|
| **难度** | Access：**低**（零应用代码）。主人密码：**中**（登录流 + 中间件 + 前端登录页，逻辑清晰、范围有限）。 |
| **可行性** | **高**。两种方式在 Cloudflare + Vue 下都成熟可用。 |
| **数据迁移** | **无**。KV 结构不变。 |
| **对现有功能的影响** | 仅“谁能调用 API”；书签/分类/AI 等业务逻辑不变。 |

## 三、多租户方案

### 3.1 数据与存储

- **KV 结构必须调整**：所有与“用户数据”相关的 key 都要按用户分片，否则无法隔离。  
  - 当前：`bookmarks`、`categories`、`pinned`、`settings` 各一份。  
  - 多租户示例：`bookmarks:${userId}`、`categories:${userId}`、`pinned:${userId}`、`settings:${userId}`。  
- **userId 从哪来**：登录后写入 session（cookie 或 JWT），每次请求从 session 中解析出 userId，再拼 key。

### 3.2 认证与用户身份

- **必须有“登录/注册”**：  
  - 要么：第三方登录（如 Google、GitHub OAuth），由 Cloudflare Access 或 Worker 做 OAuth 回调，得到邮箱/唯一 id 作为 userId。  
  - 要么：自建账号（邮箱+密码），用户表存 D1 或 KV（如 `user:${email}` → userId + 密码 hash），登录后发 session。
- **每次请求**：鉴权中间件先解析 cookie/JWT，得到 userId；再在后续 handler 里用该 userId 读写的 KV key。

### 3.3 需要改动的部分

- **后端**  
  - **鉴权**：登录/注册接口（或 OAuth 回调）、logout；所有读写数据的接口统一经过“解析 userId”的中间件。  
  - **KV 调用处**：凡使用 `getJson(kv, 'bookmarks')`、`kv.put('bookmarks', ...)` 等的地方，改为 `getJson(kv, \`bookmarks:${userId}\`)`、`kv.put(\`bookmarks:${userId}\`, ...)`；对 `categories`、`pinned`、`settings` 同理。  
  - **admin 接口**：如 cleanup-uncategorized，也要带 userId，只清理该用户的未分类数据。  
  - **AI**：当前 AI 配置在 `settings` 里，多租户下自然变成“每用户一份 settings”，无需额外设计。
- **前端**  
  - 登录/注册页（或“用 Google 登录”等）；登录状态展示（如“当前用户：xxx”）；退出登录。  
  - 所有调用 `/api/data/*`、`/api/admin/*`、`/api/ai/*` 的请求需携带 session（cookie 自动；若 JWT 则 header）。  
  - 401 时跳转登录或提示重新登录。
- **数据迁移（若已有线上单用户数据）**  
  - 方案 A：定义“默认用户”（如固定 userId = `legacy` 或首个注册用户），把现有 `bookmarks`、`categories` 等 key 复制或重命名为 `bookmarks:legacy` 等，并在登录逻辑里对“首次/无用户”做兼容。  
  - 方案 B：部署多租户前清空或导出旧数据，新部署后所有用户从零开始。  
  - 迁移脚本可一次性跑（如用 wrangler kv 或 Worker 内逻辑），只需考虑一次。

### 3.4 难度与可行性

| 项目 | 评估 |
|------|------|
| **难度** | **中高**。涉及：用户模型与登录方式、所有 KV 读写的 key 改造、中间件统一注入 userId、前端登录/注册与状态、以及（可选）数据迁移。 |
| **可行性** | **高**。Cloudflare 上可用 Workers + KV/D1、Access 的 OAuth、或自建 JWT；Vue 端登录态与请求携带方式都是常规做法。 |
| **数据迁移** | 若现有单用户数据要保留，需要**一次性迁移**（把全局 key 映射到某一 userId）；若可接受“新环境从零开始”则无迁移。 |
| **对现有功能的影响** | 业务逻辑（书签 CRUD、分类、AI、admin 清理）不变，但**所有**与 KV 相关的路径都要多一个“按 userId 取 key”的维度，改动面大、需逐处检查。 |

## 四、对比小结

| 对比项 | 单用户（对外暴露） | 多租户 |
|--------|--------------------|--------|
| **KV 结构** | 不改，继续全局 key | 所有数据 key 加 `:${userId}` 前缀 |
| **认证** | 门禁（Access）或主人密码；无“账号体系” | 必须有登录/注册或 OAuth，有 userId |
| **后端改动范围** | 仅“是否做应用内登录+鉴权中间件”（Access 则零改动） | 鉴权 + **所有** data/admin/ai 的 KV 读写处都要带 userId |
| **前端改动范围** | 仅“主人密码”时：登录页 + 401 处理 + 请求带 session | 登录/注册页、登录状态、退出、所有 API 请求带 session |
| **数据迁移** | 无 | 可选：旧数据迁移到某一 userId |
| **难度** | 低（Access）或 中（主人密码） | 中高 |
| **可行性** | 高 | 高，但工作量和回归面更大 |

---

## 五、如何选择

- **仅自己（或极少数可信的人）用、且可接受用 Access 或一个密码“锁整站”**：选**单用户**即可，实现快、风险小。  
- **需要多人独立使用、每人一套书签且互不可见**：必须走**多租户**，接受较大改动和迁移设计。  
- **先单用户、以后可能多租户**：单用户实现时避免在业务逻辑里写死“全局 key”（例如把 key 生成抽成函数），日后多租户时只需改 key 生成与中间件注入 userId，可略微减轻后续改造量。
